## 山竹目标任务模块设计文档

### 1. 模块定位

目标任务模块是山竹系统的核心业务模块，用于承载“数据化管理自我”的主链路。

本模块解决的问题是：

- 我长期要达成什么目标？
- 这个目标应该拆成哪些子目标？
- 每个子目标下面有哪些具体任务？
- 当前目标推进到哪里？
- 哪些任务真正推动了目标？
- 我的精力长期投向了哪些分类？
- 我在什么场景、什么状态下适合做哪些任务？

模块核心链路：

```text
创建目标
  -> 拆子目标
  -> 给子目标拆任务
  -> 执行任务
  -> 更新子目标进度
  -> 汇总目标进度
  -> 记录进展
  -> 完成阶段
  -> 完成目标
  -> 进入复盘和统计
```

---

## 2. 核心概念

### 2.1 目标 `ShanzhuGoal`

目标代表一个长期或阶段性的结果。

目标回答的是：

> 我要达成什么结果？

示例：

- 3 个月完成山竹系统 MVP
- 今年读完 24 本书
- 90 天减重 8 斤
- 每月稳定存下 5000 元

目标不应该是每天的小待办，而应该是一个相对稳定的方向。

---

### 2.2 子目标 `ShanzhuSubGoal`

子目标代表目标下面的阶段、里程碑或模块拆解。

子目标回答的是：

> 我要分几步完成这个目标？

示例：

目标：

```text
完成山竹系统 MVP
```

子目标：

- 完成业务设计
- 完成目标管理模块
- 完成任务管理模块
- 完成习惯追踪模块
- 完成复盘模块
- 完成仪表盘和整体联调

子目标不是每天的待办，而是中间成果。

---

### 2.3 任务 `ShanzhuTask`

任务代表具体执行动作。

任务回答的是：

> 我今天或本周具体要做什么？

示例：

子目标：

```text
完成目标管理模块
```

任务：

- 设计 `shanzhu_goal` 表
- 设计 `shanzhu_sub_goal` 表
- 设计 `shanzhu_task` 表
- 实现目标新增接口
- 实现目标分页接口
- 实现目标详情页
- 实现目标列表页

任务是执行层，应该可以频繁新增、完成、延期、取消。

---

## 3. 目标、子目标、任务关系

### 3.1 推荐模型

```text
ShanzhuGoal
  ├── ShanzhuSubGoal
  │     └── ShanzhuTask
  ├── ShanzhuSubGoal
  │     └── ShanzhuTask
  ├── 未归属子目标的 ShanzhuTask
  └── ShanzhuGoalProgress
```

一句话：

> 目标负责方向，子目标负责拆解，任务负责执行。

---

### 3.2 为什么不做目标直接绑定任务

不推荐只做：

```text
目标
  └── 任务
```

原因：

- 目标详情页容易变成杂乱待办池。
- 任务粒度大小不一致，直接按任务数算目标进度会失真。
- 目标是结果，任务是动作，两者层级不同。
- 缺少中间阶段后，很难看清目标的完成路径。

---

### 3.3 为什么不做递归子目标

第一期不推荐做：

```text
目标
  └── 子目标
       └── 子目标
            └── 任务
```

原因：

- 递归结构实现复杂。
- 页面交互复杂。
- 进度计算复杂。
- 第一版容易过度设计。

第一期固定三层即可：

```text
目标 -> 子目标 -> 任务
```

---

## 4. 分类和标签设计

分类和标签都要设计，但二者定位不同。

```text
分类 = 人生领域 / 统计归属
标签 = 场景特征 / 灵活筛选
```

---

### 4.1 分类 `ShanzhuCategory`

分类是稳定的、少量的、偏统计的。

分类回答的是：

> 这件事属于哪个长期投入领域？

分类主要用于：

- 仪表盘统计
- 目标分组
- 完成率统计
- 时间投入统计
- 不同领域平衡分析

---

### 4.2 标签 `ShanzhuTag`

标签是灵活的、多选的、偏检索和上下文的。

标签回答的是：

> 这件事有什么属性、场景、状态或备注？

标签主要用于：

- 快速筛选
- 任务中心过滤
- 今日计划编排
- 复盘时查找
- 标记特殊状态

---

### 4.3 分类和标签使用规则

```text
目标：必须有一个分类，可以有多个标签
子目标：继承目标分类，可以有多个标签
任务：继承目标分类，可以有多个标签
```

详细规则：

- `ShanzhuGoal` 保存 `category_id`。
- `ShanzhuSubGoal` 不单独保存分类，默认继承目标分类。
- `ShanzhuTask` 不单独保存分类，默认继承目标分类。
- 目标、子目标、任务都可以绑定多个标签。
- 分类用于稳定统计，标签用于灵活筛选。

---

## 5. 第一版内置分类

第一版分类先使用内置分类，不做复杂分类管理页面。

| 分类 | 说明 |
|---|---|
| 项目建设 | 山竹系统、个人项目、产品开发 |
| 成长学习 | 读书、课程、技能提升 |
| 健康运动 | 运动、减脂、睡眠、饮食 |
| 工作事业 | 工作目标、职业发展 |
| 财务资产 | 存钱、理财、收入提升 |
| 生活习惯 | 作息、整理、日常规律 |
| 关系家庭 | 家庭、朋友、亲密关系 |
| 个人管理 | 时间管理、精力管理、自我系统 |
| 其他 | 无法归类的目标 |

---

## 6. 标签类型设计

标签本身可以有类型，但第一期不需要让用户感知太复杂。

### 6.1 场景标签

用于描述任务适合在什么场景做：

- 需要电脑
- 手机可做
- 外出
- 家里
- 公司
- 碎片时间
- 安静环境

---

### 6.2 能量标签

用于描述任务需要多少精力：

- 高能量
- 中能量
- 低能量

示例：

```text
低能量 + 30分钟内 + 手机可做
```

适合用户在疲惫时筛选任务。

---

### 6.3 时间标签

## 9. 进度计算设计

### 9.1 进度模式

目标支持三种进度模式：

| 模式 | 含义 |
|---|---|
| `manual` | 手动维护目标进度 |
| `sub_goal` | 根据子目标汇总 |
| `task` | 根据任务辅助汇总 |

---

### 9.2 推荐第一期进度规则

第一期采用：

```text
目标进度 = 子目标进度汇总
子目标进度 = 任务完成率参考 + 用户手动修正
任务 = 完成 / 未完成
```

也就是说：

- 系统给自动参考值。
- 用户可以手动调整。
- 最终以用户确认的进度为准。

---

### 9.3 目标进度计算

如果子目标有权重：

```text
目标进度 = Σ 子目标进度 × 子目标权重
```

如果子目标没有设置权重：

```text
目标进度 = 所有子目标进度平均值
```

---

### 9.4 子目标进度计算

子目标可以参考任务完成率：

```text
子目标参考进度 = 已完成任务数 / 总任务数 × 100
```

注意：

> 任务大小不同，所以任务完成率只能作为参考，不能强制等于子目标最终进度。

## 17. 关键约束

- 后端命名统一使用 `ShanzhuXxx`。
- 接口统一使用 `/shanzhu/...`。
- 目标必须选择分类。
- 子目标和任务默认继承目标分类。
- 标签是多选关系，不能直接用单字段字符串硬塞。
- 删除统一逻辑删除。
- 状态字段和标签字段要分离，标签不能替代状态。
- 进度计算要允许用户手动修正。
- 第一版不做递归子目标。
- 第一版先把目标、子目标、任务主链路打通。

## 18. 最终结论

山竹目标任务模块最终采用：

```text
目标 ShanzhuGoal
  ├── 子目标 ShanzhuSubGoal
  │     └── 任务 ShanzhuTask
  ├── 未归属子目标的任务 ShanzhuTask
  └── 进展记录 ShanzhuGoalProgress
```

分类和标签采用：

```text
分类：目标必须选择，子目标和任务继承目标分类
标签：目标、子目标、任务都可多选
```

核心价值：

> 分类用于看清长期投入方向，标签用于在执行时快速找到合适的事情。

这套设计既能支撑第一期快速开发，也能为后续仪表盘、复盘、习惯、统计分析留下足够扩展空间。
