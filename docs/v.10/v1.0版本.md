## 一、系统定位

这个系统不只是一个 `Todo List`，也不只是一个目标管理后台。

它的最终定位应该是：

> **个人数据驱动自我成长系统**

它要帮助用户持续回答几个问题：

- **我想去哪里？**
- **我现在在推进什么？**
- **我每天实际做了什么？**
- **我的行为是否稳定？**
- **我的时间和精力投向了哪里？**
- **我下一步应该怎么调整？**

所以系统不是简单围绕“任务完成”展开，而是围绕一个完整闭环：

```text
方向 / 领域
  ↓
目标
  ↓
任务 + 习惯
  ↓
进展记录 + 打卡记录
  ↓
仪表盘 + 复盘
  ↓
调整目标 / 任务 / 习惯
```

---

## 二、核心设计原则

### 1. 目标是主轴，但不是唯一入口

目标很重要，因为目标定义了结果。

但系统不能变成所有东西都强制挂在目标下面。

更合理的是：

```text
目标是结果主线
任务是一次性行动
习惯是持续性行动
进展是过程记录
复盘是跨模块分析
分类 / 领域是人生方向维度
```

---

### 2. 习惯和目标分开建模

习惯不应该做成目标的子目标，也不应该做成循环任务。

习惯应该是独立对象，但可以关联目标。

```text
习惯可以独立存在
习惯也可以关联目标
习惯也可以关联子目标
```

例如：

```text
习惯：每天喝水
关联目标：无

习惯：每周跑步 3 次
关联目标：改善身体状态

习惯：每天写 300 字
关联目标：提升写作能力
```

---

### 3. 任务和习惯不要混在一起

任务是：

```text
做完就结束的一次性动作
```

习惯是：

```text
长期重复的持续行为
```

所以第一版不要把习惯每天自动生成任务，否则会导致任务中心混乱。

正确方式是：

```text
习惯规则 + 打卡记录
```

---

### 4. Web 和小程序分工明确

当前先做 Web 和后端能力，小程序作为后续低摩擦输入端。

```text
Web：
- 目标规划
- 任务管理
- 习惯管理
- 数据仪表盘
- 深度复盘
- 长篇总结

微信小程序：
- 今日任务
- 今日打卡
- 快速记录
- 快速进展
```

一句话：

> **Web 是大脑，小程序是手脚。**

## 六、习惯与打卡模块详细设计

### 1. 习惯是什么？

习惯是一个持续性行为规则。

例如：

```text
每天阅读 30 分钟
每周跑步 3 次
每天写复盘
每周输出一篇文章
每月整理一次财务
```

它关注的不是一次完成，而是：

```text
频率
稳定性
完成率
连续性
长期积累
```

---

### 2. 习惯表设计

表名建议：

```text
shanzhu_habit
```

字段建议：

```text
id
user_id
goal_id
sub_goal_id
title
description
frequency_type
target_count
target_value
unit
start_date
end_date
status
sort_order
remark
del_flag
create_time
update_time
```

字段说明：

- **goal_id**：可选，关联目标
- **sub_goal_id**：可选，关联子目标
- **title**：习惯名称
- **frequency_type**：频率类型
- **target_count**：每周期目标次数
- **target_value**：每次目标值
- **unit**：单位
- **status**：状态

---

### 3. 习惯频率

第一版只支持：

```text
daily    每天
weekly   每周
monthly  每月
```

不做复杂 Cron，不做节假日规则，不做自定义复杂周期。

---

### 4. 习惯状态

```text
active      进行中
paused      已暂停
completed   已结束
cancelled   已取消
```

---

### 5. 打卡记录表设计

表名建议：

```text
shanzhu_habit_checkin
```

字段建议：

```text
id
user_id
habit_id
goal_id
sub_goal_id
checkin_date
actual_value
unit
note
mood
status
del_flag
create_time
update_time
```

说明：

- **habit_id**：关联习惯
- **goal_id / sub_goal_id**：冗余保存，方便历史统计
- **checkin_date**：打卡日期
- **actual_value**：实际完成值
- **note**：打卡备注
- **mood**：状态感受，可选
- **status**：打卡状态

第一版建议唯一约束：

```text
user_id + habit_id + checkin_date
```

即一个用户同一个习惯同一天最多一条有效打卡记录。

## 八、统计规则设计

### 1. 今日是否需要打卡

#### daily

每天都需要打卡。

#### weekly

当前周内需要完成 `target_count` 次。

例如：

```text
每周跑步 3 次
```

本周打卡满 3 次即达标。

#### monthly

当前月内需要完成 `target_count` 次。

例如：

```text
每月读书 4 次
```

本月打卡满 4 次即达标。

---

### 2. 本周期完成率

公式：

```text
完成率 = 当前周期已打卡次数 / 当前周期目标次数 * 100
```

例如：

```text
每周跑步 3 次
本周已跑 2 次
完成率 = 2 / 3 * 100 = 66.67%
```

---

### 3. 连续打卡天数

第一版只对 `daily` 类型计算连续天数。

规则：

- 今天已打卡：从今天往前算
- 今天未打卡但昨天打卡：从昨天往前算
- 遇到未打卡日期中断

`weekly` 和 `monthly` 第一版可以先不计算连续天数，后续再扩展为连续达标周期数。

## 十五、最终结论

当前系统已经完成了：

```text
目标
子目标
任务
进展
仪表盘
复盘
分类 / 标签辅助能力
```

下一步最应该做的是：

> **新增习惯与打卡模块。**

它会补齐当前系统缺少的持续行为数据，让系统从：

```text
目标任务管理系统
```

升级为：

```text
目标 + 任务 + 习惯 + 复盘的个人数据驱动系统
```

最终结构应该是：

```text
目标定义方向
任务推动事项
习惯沉淀行为
记录形成事实
复盘产生洞察
数据驱动调整
```

一句话总结：

> **现在不是继续堆页面，而是补齐“习惯打卡”这个行为数据入口，让系统真正开始从结果管理走向行为驱动。**
